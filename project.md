# JobPilot 项目总手册（重写版）

## 1. 项目定义

### 1.1 项目名称
- 中文名：求职投递 Copilot
- 英文名：JobPilot

### 1.2 正确定位
JobPilot 的正确定位不是“求职管理 CRUD 系统”，也不是“会聊天的求职助手”，而是：

**一个面向求职场景的 Agentic AI Workflow 系统。**

它围绕用户在求职过程中的真实任务，构建一条由 AI 增强的多步骤工作流：

**JD 理解 → 简历匹配 → 材料生成 → 投递跟踪 → 面试准备 → 反馈评测**

### 1.3 核心判断
在任何阶段，都必须区分两件事：

- **项目整体目标**：构建面向求职场景的 Agentic AI Workflow 系统
- **当前实现阶段**：先完成业务数据底座、服务接口和最小闭环，再逐步接入 AI 能力

当前已完成的后端表结构和 API，只是这个项目的**底座**，不是项目的最终定义。

## 2. 项目要解决的真实问题

### 2.1 用户问题
求职者在真实投递流程中通常会遇到这些问题：

1. 看不懂岗位 JD，到底适不适合自己
2. 不清楚自己的简历与岗位之间的差距
3. 不知道某个岗位值不值得投
4. 不知道如何针对岗位改简历
5. 不知道如何生成更贴岗位的材料
6. 投递多了以后记录混乱，跟进效率低
7. 面试前不知道如何准备
8. 经验与反馈无法沉淀，下一次仍然重复低效决策

### 2.2 JobPilot 的产品目标
JobPilot 的目标不是“回答求职问题”，而是帮助用户完成一整条求职决策与执行流程：

- 帮用户理解岗位
- 帮用户判断值不值得投
- 帮用户分析简历与岗位差距
- 帮用户生成定制化材料
- 帮用户记录投递与跟进状态
- 帮用户准备面试
- 帮用户沉淀反馈并持续优化决策质量

## 3. 为什么它必须是 AI 应用研发导向项目

这个项目最终服务的求职方向，不是普通后端开发岗，而是更偏 **AI 应用研发工程师**。因此，JobPilot 必须体现的能力，不只是 Web 开发，而是以下 6 类能力：

### 3.1 Context
系统要能管理真正有业务意义的上下文，而不是只看当前一轮输入。

### 3.2 Memory
系统要能保存用户偏好、历史版本、历史投递、面试反馈、采纳情况等长期状态。

### 3.3 Tool Use
系统能力必须以工具形态拆分，而不是全靠大模型直接生成。

### 3.4 RAG
系统后续应支持检索岗位、公司、面试题、历史材料等外部知识，为生成与判断提供增强。

### 3.5 Eval
系统应有明确评测机制，而不是仅凭主观感觉调整 prompt 或输出逻辑。

### 3.6 Workflow
系统应该沿着可追踪、可恢复、可迭代的业务流程推进，而不是停留在单轮聊天。

**结论：JobPilot 的总目标必须始终围绕 Context、Memory、Tool Use、RAG、Eval、Workflow 六个关键词展开。**

## 4. 产品级目标架构

JobPilot 的整体架构应该从一开始就朝下面这个方向对齐，即使不是所有层都需要在第一阶段实现。

### 4.1 用户交互层
负责接收输入和展示输出。

输入包括：
- 简历文本 / 文件
- JD 文本 / 链接
- 用户偏好
- 历史投递记录
- 面试反馈

输出包括：
- 匹配分析结果
- 定制化材料
- 投递记录
- 面试准备内容
- 评测与反馈结果

### 4.2 Agent Orchestrator
这是系统的大脑，不负责“闲聊”，而负责：

- 判断当前用户所处的求职阶段
- 决定下一步该调用哪个工具
- 汇总工具输出
- 把结果写入状态
- 决定下一步动作

它的角色应该被定义为：

**工作流编排器（Workflow Orchestrator）**

而不是普通对话入口。

### 4.3 Tool Registry
系统能力应被拆成明确工具，而不是写成混乱的 prompt 逻辑。推荐工具包括：

- Resume Parser Tool
- JD Parser Tool
- Match Scorer Tool
- Resume Tailor Tool
- Cover Letter Tool
- Application Tracker Tool
- Interview Prep Tool
- Knowledge Retriever Tool

工具设计原则：
- 输入输出结构化
- 单一职责
- 可复用
- 可持久化结果

### 4.4 Context & Memory
系统需要保存真正有业务价值的长期状态，而不是“聊天记录堆积”。

推荐内容包括：
- 用户目标岗位
- 城市偏好
- 技术栈偏好
- 简历版本历史
- 投递记录
- 面试反馈
- 用户对生成结果的采纳情况

### 4.5 Evaluation Center
系统必须具备结果评估能力。推荐评估范围包括：

- 岗位匹配结果是否合理
- 简历修改建议是否具体且贴岗位
- 生成内容是否被采纳
- 面试题是否相关且覆盖重点

### 4.6 Data Flywheel
系统应收集用户行为反馈，用于持续优化：

- prompt
- 检索
- 工具逻辑
- 评分逻辑
- 个性化偏好

## 5. 技术架构总览

### 5.1 前端
推荐前端技术栈：
- Vue 3
- Vite
- Vue Router
- Axios
- Element Plus
- 可选：Pinia

前端的职责是承接工作流输入、展示、编辑与跟踪，而不是做很重的前端工程。

### 5.2 后端
后端推荐技术栈：
- FastAPI
- Pydantic Settings
- SQLAlchemy Async
- Alembic
- asyncpg
- uv

后端职责：
- 提供工作流承载层
- 管理业务状态
- 提供工具调用入口
- 持久化结构化结果
- 承接评测与反馈

### 5.3 基础设施与数据层
推荐技术栈：
- Docker Compose
- PostgreSQL
- pgvector
- Redis

职责划分：
- PostgreSQL：业务数据和结构化状态
- pgvector：后续检索增强
- Redis：缓存、异步任务、未来提醒与短期状态

### 5.4 AI / Workflow 层
推荐演进方向：
- 原生 function/tool calling
- LangGraph（后期）
- Prompt orchestration
- Tool routing
- Memory management
- Retrieval augmentation
- Eval pipeline

注意：
**这一层是项目的最终目标层，不是当前必须全部实现的层。**

### 5.5 交付与文档层
- Markdown / PDF / DOCX 导出
- README
- 演示脚本
- 简历表述
- 项目讲解稿

## 6. 数据模型总目标

### 6.1 核心业务表
这些表构成求职流程的核心业务状态：

- `user_profile`
- `resume`
- `resume_version`
- `job_posting`
- `match_result`
- `application_record`
- `application_event`
- `generated_artifact`

### 6.2 AI 能力表
这些表体现 AI 应用研发能力，不一定在第一阶段全部实现，但必须纳入总设计：

- `knowledge_document`
- `knowledge_chunk`
- `workflow_trace`
- `eval_run`
- `feedback_event`

### 6.3 关键表的职责

#### resume
保存：
- 简历原文
- 来源信息
- 解析结果
- 状态信息
- hash / 版本依据

#### resume_version
保存：
- 原始简历与定制版简历
- 针对不同岗位的修改结果
- 版本之间的对照

#### job_posting
保存：
- 岗位原文
- 岗位结构化解析结果
- 岗位状态

#### match_result
保存：
- 简历与岗位之间的匹配得分
- 优势
- 缺失项
- 建议

#### application_record
保存：
- 投递阶段
- 下一步动作
- 跟进信息
- 备注

#### generated_artifact
保存：
- 定制版简历
- 求职信
- 面试回答提纲
- 其他生成产物

#### workflow_trace
保存：
- 一次工作流执行的步骤
- 工具调用顺序
- 输入输出摘要
- 耗时
- 失败与重试

#### eval_run
保存：
- 评测版本
- 样本集
- rubric
- 指标结果
- 结果对比

#### feedback_event
保存：
- 用户是否采纳
- 是否修改
- 拒绝原因
- 反馈时间

## 7. 当前实现与最终目标的关系

这里是整个项目最容易跑偏的地方，必须明确。

### 7.1 当前实现的意义
当前已经完成的这些内容，例如：
- PostgreSQL / Redis
- FastAPI
- SQLAlchemy / Alembic
- Resume / JobPosting / MatchResult / ApplicationRecord API

它们的正确定位不是“项目最终成果”，而是：

**JobPilot 的业务数据底座与工作流承载层。**

也就是说，当前代码的作用是：
- 结构化求职流程中的核心对象
- 为后续 AI 输出找到持久化落点
- 为工具调用与状态流转提供数据基础
- 为评测与反馈闭环预留接口

### 7.2 错误理解
错误理解是：
> JobPilot 就是一个简历 / 岗位 / 匹配 / 投递记录系统

### 7.3 正确理解
正确理解是：
> JobPilot 是一个面向求职场景的 Agentic AI Workflow 系统；当前阶段先完成了它的后端数据底座和核心 API 承载层。

## 9. 项目边界

### 9.1 必须有
为了保证项目能体现 AI 应用研发能力，最终必须具备：
- 主工作流
- 工具调用
- 状态存储
- 检索增强（RAG）
- 评测
- 反馈闭环

### 9.2 可以后置
这些不是当前阶段最优先：
- 多 Agent
- 浏览器自动化
- 实时语音面试
- 花哨前端
- 复杂推荐系统

### 9.3 当前阶段故意不做是合理的
在当前阶段，不做下面这些并不代表项目设计有问题：
- 认证登录
- 文件上传
- 对象存储
- 自动提醒
- 状态机引擎
- LangChain / LangGraph
- 向量检索真实落地
- 多 Agent

只要项目总目标没有偏，这些都可以是后续演进内容。

## 11. 项目原则

### 原则 1：总目标不能被当前阶段覆盖
当前实现永远只是项目的一部分，不能反过来定义项目整体。

### 原则 2：先完成数据与工作流主线，再接 AI
AI 应增强主链路，而不是替代主链路。

### 原则 3：优先做可运行、可讲解、可迭代的东西
不要为了“高级感”引入不必要复杂度。

### 原则 4：所有新增功能都要问一句
它是在增强 JobPilot 作为 Agentic AI Workflow 系统的主线，还是让项目发散？

### 原则 5：项目最终比拼的是能力对齐，不是功能数量
目标是对齐 AI 应用研发能力，而不是堆功能。

## 12. 最终总结

JobPilot 的正确总定义是：

> **一个面向求职场景的 Agentic AI Workflow 系统。**

它的核心不是聊天，不是纯 CRUD，也不是单点工具，而是：

- 以求职流程为主线
- 以工具调用与状态存储为骨架
- 以检索增强、评测和反馈闭环为 AI 能力目标
- 先做业务数据底座与后端承载层，再逐步演进成完整 AI 应用

这份手册是项目从始至终的北极星文档。
之后所有功能、架构、技术选择，都应该回到这里判断：

**它是否让 JobPilot 更接近“面向求职场景的 Agentic AI Workflow 系统”？**
