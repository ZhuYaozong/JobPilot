# JobPilot 项目总手册（岗位对齐修订版）

## 1. 项目定义

### 1.1 项目名称
- 中文名：求职投递 Copilot
- 英文名：JobPilot

### 1.2 正确定位
JobPilot 的正确定位不是“求职管理 CRUD 系统”，也不是“只有一个聊天框的求职助手”，而是：

**一个面向求职场景的 Agentic AI Workflow 系统。**

它围绕用户在求职过程中的真实任务，构建一条由 AI 增强的多步骤工作流：

**JD 理解 → 简历理解 → 匹配分析 → 材料生成 → 投递跟踪 → 面试准备 → 反馈评测**

### 1.3 核心判断
在任何阶段，都必须区分两件事：

- **项目整体目标**：构建面向求职场景的 Agentic AI Workflow 系统
- **当前实现阶段**：先完成业务数据底座、服务接口和最小闭环，再逐步接入 Agent、RAG、Eval 与观测能力

当前已经完成的后端表结构和 API，只是这个项目的**业务承载底座**，不是项目的最终定义。

### 1.4 目标形态
JobPilot 的最终形态应同时具备两层能力：

1. **Workflow Workspace**  
   承接岗位、简历、匹配、材料、投递与反馈这些确定性业务对象和状态流转。

2. **AI Copilot Layer**  
   承接后续聊天 Agent、LangChain / LangGraph 编排、RAG 检索增强、记忆管理、评测与反思纠错能力。

结论不是二选一，而是：

> **工作流是骨架，Agent 是大脑，评测与观测是闭环。**

## 2. 项目要解决的真实问题

### 2.1 用户问题
求职者在真实投递流程中通常会遇到这些问题：

1. 看不懂岗位 JD，到底适不适合自己
2. 不清楚自己的简历与岗位之间的差距
3. 不知道某个岗位值不值得投
4. 不知道如何针对岗位改简历
5. 不知道如何生成更贴岗位的材料
6. 投递多了以后记录混乱，跟进效率低
7. 面试前不知道如何准备
8. 经验与反馈无法沉淀，下一次仍然重复低效决策

### 2.2 JobPilot 的产品目标
JobPilot 的目标不是“回答求职问题”，而是帮助用户完成一整条求职决策与执行流程：

- 帮用户理解岗位
- 帮用户判断值不值得投
- 帮用户分析简历与岗位差距
- 帮用户生成定制化材料
- 帮用户记录投递与跟进状态
- 帮用户准备面试
- 帮用户沉淀反馈并持续优化决策质量

### 2.3 项目的产品闭环
JobPilot 要形成的闭环不是单次对话，而是：

**理解问题 → 调用工具 → 产出结果 → 写入状态 → 收集反馈 → 评测回放 → 继续优化**

## 3. 为什么它必须是 AI 应用研发导向项目

这个项目最终服务的求职方向，不是普通后端开发岗，而是更偏 **AI 应用研发工程师**。因此，JobPilot 必须体现的能力，不只是 Web 开发，而是以下 6 类能力：

### 3.1 Context
系统要能管理真正有业务意义的上下文，而不是只看当前一轮输入。

### 3.2 Memory
系统要能保存用户偏好、历史版本、历史投递、面试反馈、采纳情况等长期状态。

### 3.3 Tool Use
系统能力必须以工具形态拆分，而不是全靠大模型直接生成。

### 3.4 RAG
系统后续应支持检索岗位、公司、面试题、历史材料等外部知识，为生成与判断提供增强。

### 3.5 Eval
系统应有明确评测机制，而不是仅凭主观感觉调整 prompt 或输出逻辑。

### 3.6 Workflow
系统应该沿着可追踪、可恢复、可迭代的业务流程推进，而不是停留在单轮聊天。

**结论：JobPilot 的总目标必须始终围绕 Context、Memory、Tool Use、RAG、Eval、Workflow 六个关键词展开。**

## 4. 与目标岗位要求的能力映射

为了确保项目真正服务于 AI 应用研发岗位，JobPilot 需要与岗位要求形成清晰映射，而不是只停留在“做了一个项目”。

### 4.1 需求理解与归因
岗位要求强调把模糊业务痛点转成明确 AI 目标。  
JobPilot 对应的落点是：

- 把“不会投、乱投、低效投”的求职痛点，拆成 JD 理解、简历理解、匹配分析、材料生成、投递跟踪、面试准备和反馈评测几个明确子问题
- 把求职场景从泛泛对话，转成可建模、可落库、可追踪的业务对象与工作流

### 4.2 架构设计
岗位要求强调 AI 原生系统架构设计，以及 Agent 核心模块规划。  
JobPilot 对应的落点是：

- 采用“业务数据底座 + Workflow Orchestrator + Tool Registry + Memory + RAG + Eval + Trace”的分层架构
- 后续逐步接入 LangChain / LangGraph，用于承接工具编排、状态流转和多步推理

### 4.3 知识与环境构建
岗位要求强调 API 接入、RAG、记忆方案、上下文注入策略。  
JobPilot 对应的落点是：

- 现阶段先完成业务 API 和状态存储
- 后续构建用户私有知识与通用知识的统一检索层
- 为 Agent 提供准确、及时、最小充分的上下文，而不是简单堆聊天历史

### 4.4 核心能力实现
岗位要求强调意图识别、任务拆解、反思纠错、标准化服务。  
JobPilot 对应的落点是：

- 将 JD 解析、简历解析、匹配分析、材料生成、面试准备等能力封装成结构化工具
- 通过可扩展的 Agent Runtime 承接后续任务分解、路由、重试、校验和错误兜底

### 4.5 系统迭代与演进
岗位要求强调自动化评测与回测。  
JobPilot 对应的落点是：

- 建立评测样本、rubric、反馈采纳率和回测机制
- 通过离线评测与真实使用反馈持续优化 prompt、检索、工具和工作流逻辑

### 4.6 性能优化
岗位要求强调高并发、异步、降级与稳定性。  
JobPilot 对应的落点是：

- 后续通过异步任务、缓存、超时控制、失败重试和降级策略保证系统稳定
- 通过低侵入性 trace 与指标采集定位问题，支撑系统长期演进

## 5. 产品级目标架构

JobPilot 的整体架构应该从一开始就朝下面这个方向对齐，即使不是所有层都需要在第一阶段实现。

### 5.1 用户交互层
负责接收输入和展示输出。

输入包括：
- 简历文本 / 文件
- JD 文本 / 链接
- 用户偏好
- 历史投递记录
- 面试反馈
- 用户在 AI 助手中的追问与补充信息

输出包括：
- 匹配分析结果
- 定制化材料
- 投递记录
- 面试准备内容
- 评测与反馈结果
- Agent 执行过程摘要与引用依据

### 5.2 Workflow Workspace
这是 JobPilot 当前阶段最重要的承载层。  
它负责：

- 展示岗位、简历、匹配、材料、投递等核心对象
- 展示各对象的状态、关联关系和时间线
- 提供确定性动作入口，例如解析、生成、流转、反馈
- 为后续 Agent 工具调用提供可靠的状态落点

### 5.3 AI Copilot
这是 JobPilot 面向 AI 应用研发方向的关键扩展层。  
它不是一个“闲聊入口”，而是一个能够挂载上下文、调用工具、读取知识、返回结构化结果的智能入口。

它应负责：
- 理解当前任务属于求职流程的哪一阶段
- 自动选择或建议下一步动作
- 调用内部工具与知识检索
- 输出带依据、可追踪、可反馈的结果

### 5.4 Agent Orchestrator
这是系统的大脑，不负责“纯聊天”，而负责：

- 判断当前用户所处的求职阶段
- 决定下一步该调用哪个工具
- 汇总工具输出
- 把结果写入状态
- 决定下一步动作
- 必要时发起反思、重试或兜底路径

它的角色应该被定义为：

**工作流编排器（Workflow Orchestrator）**

而不是普通对话入口。

### 5.5 Tool Registry
系统能力应被拆成明确工具，而不是写成混乱的 prompt 逻辑。推荐工具包括：

- Resume Parser Tool
- JD Parser Tool
- Match Scorer Tool
- Resume Tailor Tool
- Cover Letter Tool
- Interview Prep Tool
- Application Tracker Tool
- Knowledge Retriever Tool
- Feedback Recorder Tool
- Eval Runner Tool

工具设计原则：
- 输入输出结构化
- 单一职责
- 可复用
- 可持久化结果
- 支持观测与错误定位

### 5.6 Context & Memory
系统需要保存真正有业务价值的长期状态，而不是“聊天记录堆积”。

推荐内容包括：
- 用户目标岗位
- 城市偏好
- 技术栈偏好
- 简历版本历史
- 投递记录
- 面试反馈
- 用户对生成结果的采纳情况
- Agent 会话上下文摘要
- 用户长期偏好与历史策略

### 5.7 Knowledge & Retrieval
系统需要为模型构建准确、及时的知识环境，而不是只靠参数记忆。

知识源可包括：
- 用户私有知识：简历版本、历史 JD、历史匹配结果、历史产物、投递记录
- 通用知识：面试题、岗位说明、公司资料、项目经验片段、方法论文档

检索目标包括：
- 为当前任务提供最小充分上下文
- 提升生成内容的针对性和一致性
- 降低幻觉与泛化输出

### 5.8 Agent Runtime & Observability
这是为了让系统真正具备工程化 AI 应用特征而增加的一层。

它需要承接：
- 会话状态
- 工具调用记录
- 检索记录
- 模型输入输出摘要
- 失败重试、超时和降级
- 链路追踪与问题归因

### 5.9 Evaluation Center
系统必须具备结果评估能力。推荐评估范围包括：

- 岗位匹配结果是否合理
- 简历修改建议是否具体且贴岗位
- 生成内容是否被采纳
- 面试题是否相关且覆盖重点
- 检索召回是否命中关键上下文
- Agent 任务拆解是否有效

### 5.10 Data Flywheel
系统应收集用户行为反馈，用于持续优化：

- prompt
- 检索
- 工具逻辑
- 评分逻辑
- 个性化偏好
- 工作流节点设计
- Agent 路由策略

## 6. 技术架构总览

### 6.1 前端
推荐前端技术栈：
- Vue 3
- Vite
- Vue Router
- Axios
- Element Plus
- 可选：Pinia

前端的职责不应被理解为“仅做页面展示”，而应被定义为：

**JobPilot 的 AI 原生交互层。**

它需要同时承接：
- Workflow Workspace：工作流工作台
- AI Copilot：Agent 入口
- Knowledge Entry：知识库 / RAG 入口
- Feedback & Eval Surface：反馈与评测结果可视化

因此，前端不能只做后台管理系统，也不能只有一个聊天框。

### 6.2 后端
后端推荐技术栈：
- FastAPI
- Pydantic Settings
- SQLAlchemy Async
- Alembic
- asyncpg
- uv

后端职责：
- 提供工作流承载层
- 管理业务状态
- 提供工具调用入口
- 持久化结构化结果
- 承接评测与反馈
- 承接后续 Agent Runtime 和 trace 记录

### 6.3 基础设施与数据层
推荐技术栈：
- Docker Compose
- PostgreSQL
- pgvector
- Redis

职责划分：
- PostgreSQL：业务数据和结构化状态
- pgvector：后续检索增强
- Redis：缓存、异步任务、未来提醒与短期状态

### 6.4 AI / Workflow 层
推荐演进方向：
- 原生 function/tool calling
- LangChain（中期）
- LangGraph（中后期）
- Prompt orchestration
- Tool routing
- Memory management
- Retrieval augmentation
- Eval pipeline
- Trace / observability pipeline

注意：

**这一层是项目的最终目标层，不是当前必须全部实现的层。**

### 6.5 性能与稳定性层
后续需要逐步补齐：
- 异步任务处理
- 请求超时与重试
- 熔断 / 降级
- 缓存
- 限流
- 长文本 / 长上下文优化
- 可观测性指标与日志采集

### 6.6 交付与文档层
- Markdown / PDF / DOCX 导出
- README
- 演示脚本
- 简历表述
- 项目讲解稿
- Case 展示
- 评测报告

## 7. 数据模型总目标

### 7.1 核心业务表
这些表构成求职流程的核心业务状态：

- `user_profile`
- `resume`
- `resume_version`
- `job_posting`
- `match_result`
- `application_record`
- `application_event`
- `generated_artifact`

### 7.2 AI 能力表
这些表体现 AI 应用研发能力，不一定在第一阶段全部实现，但必须纳入总设计：

- `knowledge_document`
- `knowledge_chunk`
- `workflow_trace`
- `agent_session`
- `tool_call_trace`
- `retrieval_trace`
- `eval_run`
- `feedback_event`

### 7.3 关键表的职责

#### resume
保存：
- 简历原文
- 来源信息
- 解析结果
- 状态信息
- hash / 版本依据

#### resume_version
保存：
- 原始简历与定制版简历
- 针对不同岗位的修改结果
- 版本之间的对照

#### job_posting
保存：
- 岗位原文
- 岗位结构化解析结果
- 岗位状态

#### match_result
保存：
- 简历与岗位之间的匹配得分
- 优势
- 缺失项
- 建议

#### application_record
保存：
- 投递阶段
- 下一步动作
- 跟进信息
- 备注

#### generated_artifact
保存：
- 定制版简历
- 求职信
- 面试回答提纲
- 其他生成产物

#### workflow_trace
保存：
- 一次工作流执行的步骤
- 工具调用顺序
- 输入输出摘要
- 耗时
- 失败与重试

#### agent_session
保存：
- 一次 AI 助手会话的上下文
- 当前挂载对象
- 会话摘要
- 历史策略信息

#### tool_call_trace
保存：
- 每次工具调用的工具名
- 输入摘要
- 输出摘要
- 耗时
- 失败信息

#### retrieval_trace
保存：
- 检索 query
- 命中文档 / chunk
- 召回分数
- 实际注入内容摘要

#### eval_run
保存：
- 评测版本
- 样本集
- rubric
- 指标结果
- 结果对比

#### feedback_event
保存：
- 用户是否采纳
- 是否修改
- 拒绝原因
- 反馈时间

## 8. 当前实现与最终目标的关系

这里是整个项目最容易跑偏的地方，必须明确。

### 8.1 当前实现的意义
当前已经完成的这些内容，例如：
- PostgreSQL / Redis
- FastAPI
- SQLAlchemy / Alembic
- Resume / JobPosting / MatchResult / ApplicationRecord API
- 工作流状态层
- AI 解析与生成接口
- 最小 feedback 记录

它们的正确定位不是“项目最终成果”，而是：

**JobPilot 的业务数据底座与工作流承载层。**

也就是说，当前代码的作用是：
- 结构化求职流程中的核心对象
- 为后续 AI 输出找到持久化落点
- 为工具调用与状态流转提供数据基础
- 为评测与反馈闭环预留接口
- 为后续 Agent / RAG / Eval / Trace 提供承接空间

### 8.2 错误理解
错误理解是：
> JobPilot 就是一个简历 / 岗位 / 匹配 / 投递记录系统

或者：
> JobPilot 就是一个会聊天的求职助手

### 8.3 正确理解
正确理解是：
> JobPilot 是一个面向求职场景的 Agentic AI Workflow 系统；当前阶段先完成了它的后端数据底座、工作流接口承载层和部分真实 AI 能力入口。

## 10. 项目边界

### 10.1 必须有
为了保证项目能体现 AI 应用研发能力，最终必须具备：
- 主工作流
- 工具调用
- 状态存储
- 检索增强（RAG）
- 评测
- 反馈闭环
- 基础观测与 trace

### 10.2 可以后置
这些不是当前阶段最优先：
- 多 Agent
- 浏览器自动化
- 实时语音面试
- 复杂推荐系统
- 大规模生产级部署

### 10.3 当前阶段故意不做是合理的
在当前阶段，不做下面这些并不代表项目设计有问题：
- 认证登录
- 文件上传
- 对象存储
- 自动提醒
- 完整状态机引擎
- LangChain / LangGraph 的正式接入
- 向量检索真实落地
- 多 Agent
- 完整可观测平台

只要项目总目标没有偏，这些都可以是后续演进内容。

## 11. 知识与环境构建方案

### 11.1 用户私有知识
后续 RAG 应优先围绕用户自己的求职资产构建，包括：
- 简历原文
- 定制版简历
- 历史 JD
- 历史匹配结果
- 历史生成产物
- 投递记录
- 面试反馈

### 11.2 通用知识
还可逐步补充：
- 面试题资料
- 公司与部门信息
- 岗位说明
- 技术栈知识
- 项目经验素材库

### 11.3 上下文注入策略
上下文不应简单拼接，而应采取分层注入：

1. 当前任务对象  
   例如当前 Resume、当前 JD、当前 MatchResult

2. 检索补充信息  
   例如历史版本、相似岗位、面试题片段

3. 用户长期偏好  
   例如城市偏好、岗位方向、技术栈倾向

4. 工作流状态  
   例如当前已经做到哪一步、上一步动作是什么、下一步推荐是什么

### 11.4 知识环境的目标
目标不是“让模型知道更多”，而是：
- 提供准确上下文
- 缩小任务空间
- 降低幻觉
- 提高输出一致性
- 为 Agent 提供可验证依据

## 12. Eval 与效果迭代机制

### 12.1 为什么必须做 Eval
如果没有评测，系统就只能依赖主观感觉调整 prompt 或流程，这不符合 AI 应用研发项目的要求。

### 12.2 可评测对象
可逐步评测：
- JD 解析质量
- Resume 解析质量
- 匹配分析合理性
- 材料生成针对性
- 面试准备相关性
- 检索命中率
- Agent 路由与工具选择质量

### 12.3 Eval 方式
建议组合使用：
- 样本集回放
- rubric 打分
- 人工审阅
- 用户采纳率
- 编辑后采用率
- 拒绝率
- 对比实验

### 12.4 迭代对象
Eval 结果应反向驱动：
- prompt 改写
- 工具逻辑优化
- 检索策略优化
- 上下文注入优化
- Agent 路由优化
- 输出结构优化

## 13. 风险控制与工程兜底

### 13.1 总原则
JobPilot 不能把关键业务流程完全交给模型自由发挥，而应遵循：

- 能结构化就结构化
- 能工具化就不自由生成
- 能校验就不盲信
- 能兜底就不裸奔

### 13.2 模型局限
需要明确承认并设计应对策略：
- 幻觉
- 指令跟随不稳定
- 长上下文漂移
- 输出格式不稳定
- 对求职场景的泛化建议过强但针对性不足

### 13.3 Prompt 注入与上下文污染
后续接入 RAG 与聊天 Agent 时，应特别注意：
- 外部文本不可直接无保护注入系统提示
- 用户上传材料与检索片段需要隔离角色
- 不应让单条恶意文本覆盖系统级工作流规则
- 重要动作需保留显式确认或确定性校验

### 13.4 失败兜底
关键流程建议保留：
- 明确的前置条件校验
- 接口级错误返回
- 超时与重试
- 人工确认点
- 降级路径
- 结果可回滚或可重跑

## 14. 前端产品策略

### 14.1 前端不是后台管理系统
前端不能只做表格、表单和详情页，否则很容易退化成普通管理台。

### 14.2 前端也不是单聊天框
前端也不能从一开始就把主入口做成“万能聊天框”，否则会弱化工作流、状态存储和可追踪性。

### 14.3 正确前端形态
前端应采用“双层结构”：

1. **Workflow Workspace**
   - 工作台
   - 岗位 JD
   - 简历
   - 匹配分析
   - AI 材料
   - 投递跟踪

2. **AI Copilot Layer**
   - AI 助手
   - 知识库 / 资料库
   - 后续 LangChain / LangGraph / RAG 接入口

### 14.4 前端的第一阶段目标
第一阶段前端应优先围绕已有后端 API 落地：
- JD 理解
- 简历理解
- 匹配分析
- 材料生成
- 投递跟踪
- 面试准备
- feedback 记录

同时要为后续这些能力预留入口：
- AI 助手
- 知识库
- RAG
- Trace / Eval 可视化

## 16. 项目原则

### 原则 1：总目标不能被当前阶段覆盖
当前实现永远只是项目的一部分，不能反过来定义项目整体。

### 原则 2：先完成数据与工作流主线，再接 AI
AI 应增强主链路，而不是替代主链路。

### 原则 3：工作流与 Agent 要并存，不走两个极端
既不能只做管理台，也不能只做聊天框。

### 原则 4：所有 AI 能力都要有承载点、反馈点和评测点
不能只生成，不落库；不能只输出，不反馈；不能只上线，不评测。

### 原则 5：所有新增功能都要问一句
它是在增强 JobPilot 作为 Agentic AI Workflow 系统的主线，还是让项目发散？

### 原则 6：项目最终比拼的是能力对齐，不是功能数量
目标是对齐 AI 应用研发能力，而不是堆功能。

## 17. 最终总结

JobPilot 的正确总定义是：

> **一个面向求职场景的 Agentic AI Workflow 系统。**

它的核心不是聊天，不是纯 CRUD，也不是单点工具，而是：

- 以求职流程为主线
- 以工作流与状态存储为骨架
- 以工具调用与结构化结果为执行方式
- 以知识检索、评测、反馈和观测为 AI 能力目标
- 先做业务数据底座与工作流承载层，再逐步演进成完整 AI 应用
